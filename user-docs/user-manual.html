adobe-downloader

Station10 internal tool — User Manual — May 2026

Overview

adobe-downloader is a Python command-line tool that downloads, transforms, and manages Adobe Analytics report data for Station10 clients. It replaces the earlier JavaScript codebase (downloadAdobeTableLocal) and handles everything from pulling ranked reports from the Adobe Analytics API through to producing analysis-ready CSV files.

Core abstraction

Everything the tool does is driven by a YAML config file. Each config file describes one job: what data to pull, which RSIDs to query, what date range to cover, and how to transform the output. Once the config is written and validated, running the job is a single command:

adobe-downloader run --config jobs/examples/legend_bot_investigation_jan25.yaml

The tool is safe to interrupt and re-run. Every API request is tracked in a SQLite state database; completed requests are skipped automatically on resume so you never download the same file twice.

What it does end-to-end

  1. Loads and validates your YAML config against a strict Pydantic schema.
  2. Expands the job into individual API requests (one per RSID × date interval × report × segment).
  3. Downloads each request as a JSON file into <base_folder>/<client>/JSON/.
  4. Transforms each JSON file to a CSV using the configured transform type.
  5. Optionally concatenates all CSVs into a single output file and/or zips them.

Job types at a glance

job_typeWhat it does
report_downloadDownload ranked reports from the Adobe Analytics API for one or more RSIDs over a date range.
transform_concatTransform existing JSON files to CSV and concatenate them, without hitting the API.
segment_creationCreate Adobe Analytics segments from a CSV list of bot rules and share them with users.
lookup_generationBuild a dimension value → numeric item-ID lookup file used when constructing segment conditions.
rsid_updateFetch all report suites, download topline visit counts, and write filtered RSID list files.
compositeChain multiple steps (any combination of the above) into a single resumable pipeline.

Setup

Prerequisites

  • Python 3.12 or later
  • Access to the AdobeBotAnalysis GitHub repository
  • Adobe Analytics API credentials for the client you are working with

Install the package

From the repository root, install in editable mode:

pip install -e .

This installs the adobe-downloader command into your Python environment. Verify it works:

adobe-downloader --version

Credentials file

Each client needs a credentials file at:

credentials/client<ClientName>.yaml

For example, the Legend client uses credentials/clientLegend.yaml. The file contains Adobe API credentials in the following format:

adobe:
  clientID: <your-client-id>
  clientSecret: <your-client-secret>
  adobeOrgID: <your-org-id>@AdobeOrg
  globalCompanyID: <company-id>
  rsid: <a-default-rsid>

Security: The credentials/ directory is excluded from git commits. Never commit credential files. If you are setting up a new environment, copy the credentials file from the team's secure storage.

Verify credentials

To confirm your credentials are valid, list all report suites for a client:

adobe-downloader list-rsids --client Legend

To list all Adobe Analytics users available to the client:

adobe-downloader list-users --client Legend

Where to run commands from

Always run adobe-downloader from the repository root. The tool resolves data/ paths relative to the current working directory, so running from a different location will cause file-not-found errors.

Key Concepts

Jobs and YAML configs

A job is a single unit of work described by a YAML file. The job_type field at the top of the file determines what the job does and which fields are valid. Config files live in jobs/ — use jobs/templates/ as starting points and jobs/examples/ to see real worked examples for the Legend client.

RSIDs

An RSID (Report Suite ID) is Adobe's identifier for a report suite. The tool accepts RSIDs in three ways, controlled by the rsids.source field:

sourceHow RSIDs are supplied
singleOne RSID written directly in the config (rsids.single).
listA YAML list of RSIDs written in the config (rsids.list).
fileA plain-text file with one RSID per line (rsids.file). See below for file format.

RSID list files

RSID list files in data/rsid_lists/ contain one entry per line. Lines beginning with # are comments. Each non-comment line is a plain RSID clean name (the human-readable name used internally, not the raw Adobe rsid string):

# Minimum threshold = 1000
# Date range = 2025-01-01 to 2025-01-31
OnlineBettingcom
MarcadoresOnlinecom
Casinoorg

These files are generated automatically by the rsid_update flow (see RSID Update) and should not normally be edited by hand.

Report definitions

A report definition specifies what to request from the Adobe API: a dimension, a list of metrics, optional segments, and a row limit. Definitions live in report_definitions/ as YAML files and are organised into groups. You reference them in a config using either:

  • report_ref: <name> — one named report from the registry
  • report_group: <group> — all reports in a group (downloads them all)
  • report: — an inline definition written directly in the config (for one-off reports)

See Report Definitions for the full list of available groups and reports.

Date intervals

The interval field controls how the date range is split into individual API requests:

  • full — one request covering the entire date range
  • month — one request per calendar month within the range
  • day — one request per day (produces many files; use with care)

Most bot investigation work uses month to get monthly breakdowns.

State database and resume

When a report_download job starts, a SQLite database is created at:

<output.base_folder>/<client>/state/<job_id>.db

Every request is recorded as pending before it runs and updated to completed or failed afterward. If the job is interrupted, re-running the same command resumes from where it left off. Completed requests are skipped. Use --no-resume to force a fresh start.

Transform types

After downloading, JSON files are converted to CSV using a transform type that knows the structure of the downloaded data:

typeUsed for
standardGeneric ranked-report output. Suitable for most one-off reports.
bot_investigationBot investigation reports (adds botRuleName, rsid columns).
bot_validationBot validation reports with per-rule segment iteration.
bot_rule_compareSide-by-side comparison of bot rule variants.
final_bot_metricsAnnual final bot metrics summary.
summary_totalTopline totals (visitors, visits) with no dimension breakdown.

Output folder layout

All output is written under output.base_folder/<client>/:

<base_folder>/
  <client>/
    JSON/          raw downloaded JSON files
    CSV/           transformed CSV files
    state/         SQLite state databases
    config_archive/  copies of the config file saved after each run
    job_history.jsonl  append-only run history log

Running Jobs

The run command

The primary command for executing any job:

adobe-downloader run --config <path-to-config.yaml>
FlagDefaultDescription
--config / -crequiredPath to the job YAML config file.
--report / -rNoneDownload only this one named report (overrides report_group expansion).
--no-resumeFalseIgnore existing state and re-download everything from scratch.
--testFalseEnable test mode: caps RSIDs, date intervals, and segments per test_limits config. See Test Mode.

validate

Parse and schema-check a config file without running any downloads. Always run this first when writing a new config.

adobe-downloader validate --config jobs/examples/legend_bot_investigation_jan25.yaml
FlagDefaultDescription
--config / -crequiredPath to the config file to validate.
--check-credentials / --no-check-credentialsTrueWarn if the credentials file for the config's client is missing.

Exit codes: 0 = valid, 1 = schema error, 2 = valid schema but referenced files missing.

status

Print the current state of a report_download job (completed / pending / failed counts).

adobe-downloader status --config jobs/examples/legend_bot_investigation_jan25.yaml

retry

Re-queue failed (or all non-completed) requests so they will be attempted on the next run.

adobe-downloader retry --config <config.yaml> [--failed-only]
FlagDescription
--failed-onlyRe-queue only requests with status failed. Omitting this flag re-queues all non-completed requests.

reset

Delete all state for a job so it starts completely fresh. Requires --confirm.

adobe-downloader reset --config <config.yaml> --confirm

validate-output

Check that all expected output files exist and are non-empty after a download job.

adobe-downloader validate-output --config <config.yaml> [--dry-run] [--retry]
FlagDescription
--dry-runReport missing files without re-downloading.
--retryRe-download any missing or empty files immediately.

history

Show recent job runs from the append-only job history log.

adobe-downloader history --client Legend [--last 10] [--status completed] [--since 2025-06-01]

transform

Transform JSON files to CSV and optionally concatenate them, without touching the API. Useful if a transform step failed after a successful download.

adobe-downloader transform \
  --json-dir "/path/to/output/Legend/JSON" \
  --pattern "Legend_botInvestigation*.json" \
  --concat \
  --concat-output "/path/to/output.csv"

cleanup

Remove old processed files. Always requires --confirm to actually delete.

adobe-downloader cleanup \
  --client Legend \
  --older-than 30d \
  --type processed-json \
  --confirm

Valid --type values: processed-json, logs, state.

Flow: Report Download

The report_download job type is the workhorse of the tool. It iterates over RSIDs, date intervals, and optionally segments, sending one ranked-report API request per combination and saving the result as a JSON file.

Minimal example

job_type: report_download
client: Legend
description: "Download bot investigation reports for Jan 2025"

report_group: bot_investigation

rsids:
  source: file
  file: data/rsid_lists/legend_investigation.txt
  batch_size: 12

interval: month

date_range:
  from: "2025-01-01"
  to: "2025-01-31"

transform:
  enabled: true
  type: bot_investigation
  concat: true

resume: true

output:
  base_folder: "/Users/edwardgurton/Documents/Work/Adobe Downloads"

RSID sources

Choose one of the three rsids.source values:

# From a file (most common for multi-RSID jobs)
rsids:
  source: file
  file: data/rsid_lists/legend_investigation.txt
  batch_size: 12  # how many RSIDs to process in parallel

# Explicit list
rsids:
  source: list
  list:
    - trillioncoverscom
    - casinoorg

# Single RSID
rsids:
  source: single
  single: trillioncoverscom

Date intervals

date_range:
  from: "2025-01-01"
  to: "2025-03-31"

interval: month   # full | month | day

Using month with the above range produces three intervals: January, February, and March 2025. The full interval makes one request covering the entire range.

Report specification

Exactly one of report_ref, report_group, or report must be set:

# Named report from the registry
report_ref: botInvestigationMetricsByBrowser

# All reports in a group
report_group: bot_investigation

# Inline definition (for one-off or experimental reports)
report:
  name: myCustomReport
  dimension: variables/daterangeday
  row_limit: 500
  segments:
    - s3938_61bb0165a88ab931afa78e4c
  metrics:
    - metrics/visitors
    - metrics/visits
    - metrics/event3
  csv_headers: [id, day, unique_visitors, visits, raw_clickouts, fileName, fromDate, toDate]

Segment iteration

When segments is set, the tool downloads a separate copy of each report for each segment ID. This is how bot validation works — one download per bot rule segment. See Segments for all source options.

segments:
  source: segment_list_file
  file: data/segment_lists/Legend/Apr25-RoundFive.json

Transform options

transform:
  enabled: true
  type: bot_investigation   # see table in Concepts
  concat: true              # concatenate all CSVs into one file after transform

post_processing:
  delete_json_after_transform: false   # set true to save disk space
  zip_csvs_after_concat: true          # zip the concat output

File naming

Output JSON files follow this naming convention:

<client>_<reportName>[_<extra>][_DIMSEG<segmentId>_]_<fromDate>_<toDate>.json

Use file_name_extra to add a label to the filename (useful for distinguishing parallel runs):

file_name_extra: "Coverscom-FullRun"

Flow: Transform and Concatenate

The transform_concat job type transforms existing JSON files to CSV and concatenates them. No API calls are made. Use this when:

  • A transform step failed after a successful download and you want to re-run it.
  • You want to apply a different transform type to already-downloaded data.
  • You need to concatenate a subset of files matching a specific pattern.

Example

job_type: transform_concat
client: Legend
description: "Re-transform existing JSON files for bot investigation Q1 2025"

transform:
  enabled: true
  type: bot_investigation
  source_pattern: ".*botInvestigationMetricsByBrowser.*2025-01.*\.json$"
  source_folder: null    # null = default JSON folder for this client
  output_subfolder: null
  concat: true

concat:
  enabled: true
  file_pattern: ".*\.csv$"
  custom_headers: null   # override column headers by index if needed

output:
  base_folder: "/Users/edwardgurton/Documents/Work/Adobe Downloads"

When to use inline transform vs. transform_concat

If you are doing a full download, use the transform block inside a report_download config. The transform_concat job type is for standalone re-processing after the download is already complete.

Concat options

FieldTypeDescription
concat.enabledboolWhether to run concatenation after transform.
concat.file_patternstring (regex)Regular expression selecting which CSV files to include. Default: .*\.csv$.
concat.custom_headersmapOverride column headers by 1-based column index. Example: 1: "MyColumn".

Transform via CLI (no config file)

For a quick ad-hoc transform, use the transform command directly:

adobe-downloader transform \
  --json-dir "/path/to/Legend/JSON" \
  --pattern "Legend_botInvestigationMetricsByBrowser*.json" \
  --concat \
  --concat-output "/path/to/output.csv"

Flow: Segment Creation

The segment_creation job type reads a CSV list of bot rules, creates the corresponding segments in Adobe Analytics via the API, and writes three output files used by downstream validation jobs.

Example config

job_type: segment_creation
client: Legend
description: "Create Apr25 bot rule segments and share with Legend user"

segment_creation:
  input_csv: data/segment_creation_lists/Apr25-RoundFive.csv
  share_with_users:
    - "200419062"     # Adobe user ID (numeric string)
  test_mode_row: null # set to a row number (1-based) to test just one row

output:
  base_folder: "/Users/edwardgurton/Documents/Work/Adobe Downloads"

Input CSV format

The input CSV must have the following columns (header row required):

ColumnValuesDescription
Compare / ValidateCompare, Validate, Compare - Special, Validate - SpecialDetermines which output list the segment appears in.
Segment NamestringHuman-readable segment name (also used as description).
RSID Clean NamestringClean name of the report suite the segment is scoped to.
Dimension 1friendly nameFirst dimension for the segment condition (e.g. BrowserType).
Dimension 1 ItemstringValue to match for dimension 1.
Dimension 2friendly name or blankOptional second dimension condition.
Dimension 2 Itemstring or blankValue to match for dimension 2.

Dimensions that require a numeric lookup (such as BrowserType, OperatingSystem, MobileManufacturer, MonitorResolution) are automatically resolved using the lookup files in data/lookups/.

Output files produced

FileContentUsed by
segment_list_path (JSON)Array of segment IDs created this run. Used for segment iteration in subsequent report downloads.report_download with segments.source: segment_list_file
compare_list_path (JSON)Bot rules marked Compare.Bot rule comparison flows
validate_list_path (JSON)Bot rules marked Validate.Bot validation flows

Output paths are configured on the segment_creation block, not on the top-level output:

segment_creation:
  input_csv: data/segment_creation_lists/Apr25-RoundFive.csv
  compare_list_path: data/bot_compare_lists/
  validate_list_path: data/bot_rule_lists/
  segment_list_path: data/segment_lists/Legend/

share_with_users

A list of Adobe user IDs (numeric strings) who will have the created segments shared with them. Find valid IDs with:

adobe-downloader list-users --client Legend

Flow: Lookup Generation

A lookup file maps human-readable dimension values (e.g. "Chrome 120") to the numeric item IDs used internally by Adobe Analytics. These IDs are required when creating segment conditions for dimensions like Browser, BrowserType, OperatingSystem, MobileManufacturer, and MonitorResolution. Lookup files are stored in data/lookups/<dimensionName>/lookup.txt.

When do you need to regenerate a lookup?

If segment_creation reports that a value cannot be found in the lookup, the lookup file is probably stale (new dimension values have appeared since it was last generated). Regenerate it using a recent date range.

Example config

job_type: lookup_generation
client: Legend
description: "Generate browser dimension lookup for trillioncoverscom (Jan 2025)"

lookup_generation:
  dimension: variables/browser     # Adobe dimension variable string
  rsid: trillioncoverscom
  segments:
    - s3938_61bb0165a88ab931afa78e4c  # Optional: filter the lookup request
  output_file: null                   # null = auto-named in data/lookups/

date_range:
  from: "2025-01-01"
  to: "2025-01-31"

output:
  base_folder: "/Users/edwardgurton/Documents/Work/Adobe Downloads"

Searching a lookup file

To check what ID a specific value maps to (useful for debugging segment conditions):

adobe-downloader search-lookup --dimension BrowserType --value "Chrome Mobile"

If the value is not found exactly, the command shows similar entries.

Flow: RSID Update

The rsid_update flow refreshes the two RSID list files used by investigation and validation jobs. It fetches all report suites for the client, downloads a topline visit count for each, and writes filtered lists based on configurable thresholds.

Run this before starting a new investigation cycle to ensure the RSID lists reflect the current set of active report suites.

Example config

job_type: rsid_update
client: Legend
description: "Refresh Legend RSID lists — run before each new investigation"

rsid_update:
  investigation_threshold: 1000  # minimum visits to include in investigation list
  validation_threshold: 1000     # minimum visits to include in validation list
  include_virtual: false         # exclude virtual report suites (rsid prefix: vrs_)

date_range:
  from: "2025-01-01"
  to: "2025-01-31"

output:
  base_folder: data/rsid_lists/

Output files

FileContent
data/rsid_lists/botInvestigationMinThresholdVisits.txtRSIDs with visits ≥ investigation_threshold.
data/rsid_lists/botValidationRsidList.txtRSIDs with visits ≥ validation_threshold.
data/report_suite_lists/<dated>.txtFull rsid:cleanName pairs file for use by downstream lookups.

Threshold fields

FieldDefaultDescription
investigation_threshold1000Minimum visit count to include a report suite in the investigation list.
validation_threshold1000Minimum visit count to include a report suite in the validation list.
include_virtualfalseWhen true, report suites with the vrs_ prefix are included.

Exclusion list

Report suites whose clean names appear in data/rsid_lists/excludedRsidCleanNames.txt are removed from both output lists regardless of visit count. Edit this file to permanently exclude specific suites.

Quick RSID update via CLI (no config file)

You can also trigger an RSID update directly with flags:

adobe-downloader update-rsids \
  --client Legend \
  --from 2025-01-01 \
  --to 2025-01-31 \
  --investigation-threshold 1000 \
  --validation-threshold 1000

Flow: Composite Jobs

A composite job chains multiple steps into a single resumable pipeline. Each step can reference the outputs of previous steps, making it possible to run segment creation, download, validate, and transform in sequence with a single command.

Example: full bot investigation pipeline

job_type: composite
client: Legend
description: "Full Legend bot investigation: segment creation + download + validate + transform"

date_range:
  from: "2025-01-01"
  to: "2025-03-31"

output:
  base_folder: "/Users/edwardgurton/Documents/Work/Adobe Downloads"

steps:
  - step: segment_creation
    id: create_segments
    segment_creation:
      input_csv: data/segment_creation_lists/Apr25-RoundFive.csv
      share_with_users:
        - "200419062"

  - step: report_download
    id: download_investigation
    report_group: bot_investigation
    rsids:
      source: file
      file: data/rsid_lists/legend_investigation.txt
      batch_size: 12
    segments:
      source: step_output
      step_id: create_segments
      output_key: segment_list_file
    interval: month

  - step: validate_output
    id: validate
    depends_on: download_investigation
    config_ref: download_investigation
    retry: true

  - step: transform_concat
    id: transform
    depends_on: validate
    transform:
      type: bot_investigation
      concat: true

Step types

stepDescription
report_downloadDownload reports from the Adobe API.
transform_concatTransform JSON files to CSV and concatenate.
segment_creationCreate segments from a CSV list.
validate_outputCheck all expected output files exist; optionally re-download missing ones.
rsid_updateRefresh RSID list files.
dim_to_segmentsGenerate segments from dimension values automatically.
lookup_generationBuild a dimension lookup file.
bot_rule_compareRun a bot rule comparison analysis.
final_bot_metricsProduce final annual bot metrics.

depends_on

Use depends_on: <step_id> to ensure a step only runs after its dependency has completed successfully. If the dependency has not completed, the pipeline halts with an error.

Inter-step output references

Use source: step_output to wire one step's output into another step's input:

segments:
  source: step_output
  step_id: create_segments        # id of the earlier step
  output_key: segment_list_file   # key in that step's output dict

Common output keys for segment_creation steps:

output_keyContent
segment_list_filePath to the JSON file listing all created segment IDs.
compare_list_filePath to the Compare bot rules list.
validate_list_filePath to the Validate bot rules list.

Common output keys for rsid_update steps:

output_keyContent
investigation_listPath to the investigation RSID list file.
validation_listPath to the validation RSID list file.

Resume across steps

Composite jobs are fully resumable. If the pipeline is interrupted mid-way, re-running the same command skips all already-completed steps and resumes from the first incomplete one. The state for each step is stored in the same SQLite database as the overall composite job.

test_mode in composite jobs

Set test_mode: true at the top level of a composite config, or pass --test on the CLI. The test limits apply to all report_download steps in the composite.

Report definition file structure

Each file follows this structure:

group: bot_investigation
description: "Human-readable description"
transform_type: bot_investigation

defaults:
  segments:
    - s3938_66fe79408ff02713f66ed76b   # applied to all reports in group
  metrics:
    - metrics/event3
  row_limit: 500

reports:
  botInvestigationMetricsByDay:
    dimension: variables/daterangeday
    csv_headers:
      - id
      - day
      - unique_visitors
      - ...

Individual reports can override segments, metrics, and row_limit from the defaults. The csv_headers list defines the column names in the output CSV (position-mapped to the API response).

Segment list JSON format

A segment list file is a JSON array of Adobe segment ID strings:

[
  "s3938_61234abc...",
  "s3938_65432def...",
  "s3938_7abc1234..."
]

These files are written to data/segment_lists/<client>/ by the segment_creation flow.

The dim_to_segments composite step automatically generates segments from the top values of a dimension. It is used in the country investigation pipeline to create per-country segments without manual CSV input:

Test Mode

Test mode limits the number of RSIDs, date intervals, and segments processed so you can verify a config works end-to-end without triggering hundreds of API requests. Always use test mode the first time you run a new config.

Enabling test mode

Either set it in the config file:

test_mode: true
test_limits:
  max_rsids: 2
  max_date_intervals: 1
  max_segments: 2

Or pass --test on the CLI (overrides the config setting):

adobe-downloader run --config jobs/examples/legend_bot_investigation_jan25.yaml --test

Recommended test config settings

SettingRecommended test valueReason
test_limits.max_rsids2Enough to verify iteration logic without a large API bill.
test_limits.max_date_intervals1One month is sufficient to check the full data pipeline.
test_limits.max_segments2Enough to verify segment iteration.
resumefalseStart fresh on each test run so you see a clean result.
transform.concatfalseToo few files to make concatenation useful.

What test mode does not limit

Test mode only caps the iteration counts. It does not change the API endpoint, authentication, or report definitions. The requests that are sent are real Adobe Analytics API calls.

Important: Test mode limits are applied in the order RSIDs/intervals/segments are iterated. The first N items are used; there is no random sampling. Check that the first few items in your RSID file and segment list are representative before running in test mode.

Troubleshooting

Schema validation failed

Symptom: adobe-downloader run or validate prints Schema validation failed with field names and error messages.

Cause: A required field is missing, a value has the wrong type, or a mutually exclusive combination is set (e.g. both report_ref and report_group).

Fix: Read the error message carefully. The field path (e.g. rsids -> source) tells you exactly what is wrong. Run validate after every edit until it passes.

No credentials file found

Symptom: Warning: no credentials file found for client 'X', or FileNotFoundError at run time.

Fix: Ensure credentials/client<Name>.yaml exists (case-sensitive). The client name in the config must match the suffix of the credentials filename exactly.

RSID file not found

Symptom: RSID file not found: data/rsid_lists/legend_investigation.txt.

Fix: Run adobe-downloader update-rsids (or a composite job with an rsid_update step) to generate the RSID list files. Also verify you are running from the repository root.

Segment list file not found

Symptom: Segment list file not found: data/segment_lists/Legend/....

Fix: Run the segment_creation job first to generate the file, or update the segments.file path to point to an existing file.

Config has changed since the last run

Symptom: Warning on run: config has changed since the last run.

Cause: The YAML file was modified after the job's state database was created. The state is keyed to the config hash.

Fix: If the change was intentional and you want to continue the previous download with the new config, proceed normally. If you want to start completely fresh, run with --no-resume or use the reset command.

Many failed requests

Symptom: status shows a large number of failed requests.

Cause: Adobe API rate limits, temporary network issues, or invalid segment/metric IDs.

Fix:

  1. Check the last errors shown by adobe-downloader status --config <config>.
  2. If rate-limited, lower rsids.batch_size (e.g. to 6) and re-run.
  3. Run adobe-downloader retry --config <config> --failed-only to re-queue failed requests.
  4. Run adobe-downloader run --config <config> again to process the re-queued requests.

Lookup value not found during segment creation

Symptom: Segment creation errors with Value 'X' not found in lookup.

Fix:

  1. Use adobe-downloader search-lookup --dimension <dim> --value <value> to search the lookup for similar entries.
  2. If the value genuinely does not exist in the lookup, regenerate the lookup file using a recent date range (see Lookup Generation).
  3. Check spelling and case. MonitorResolution values are normalised (e.g. 1920x1080 becomes 1920 x 1080).

No RSID lookup file found in data/report_suite_lists

Symptom: segment_creation aborts with No RSID lookup file found in data/report_suite_lists.

Fix: Run rsid_update to generate a dated pairs file in data/report_suite_lists/.

validate-output finds missing files after a completed run

Symptom: validate-output reports missing JSON files even though the job showed as completed.

Fix: Run adobe-downloader validate-output --config <config> --retry to re-download missing files automatically.
